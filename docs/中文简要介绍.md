## 先从一个很小的证明说起

假设你正在写一段数学证明。你已经知道 `x = 2`，接下来想说两件事：
`x + 1 = 3`，以及 `x^2 = 4`。

在 Litex 里，你可以直接这样写：

```litex
forall x R:
    x = 2
    =>:
        x + 1 = 3
        x^2 = 4
```

这段代码想表达的东西很普通：如果 `x` 是实数，并且 `x = 2`，那么后面这两个
等式都成立。Litex 做的事情也很直接：它检查这些事实能不能从当前条件推出。

这就是 Litex 最核心的交互方式：

> 你写下一条数学事实；Litex 检查它；如果检查通过，这条事实进入 verified context。

这里的重点不是“语法更短”。重点是你不用先把证明翻译成一串 tactic 命令，也不用一开始
就去想应该引用哪个库定理。你先写你在数学上想说的下一句话。Litex 如果能验证它，这句话
就成为后面证明的一部分。

## 证明为什么会“长出来”

普通数学证明不是一次性从空白跳到结论的。人写证明时，通常是一点点建立上下文：
先有对象，再有条件，再有中间结论，最后这些结论一起支撑目标。

Litex 想保留的就是这个动作。比如：

```litex
forall r, s R:
    r + 2 * s = -1
    s = 3
    =>:
        r = (r + 2 * s) - 2 * s = -7
```

读这段时，你大概不会觉得它神秘。它说：已知 `r + 2 * s = -1`，又知道
`s = 3`，所以可以把 `r` 写成 `(r + 2 * s) - 2 * s`，再代入并算出 `-7`。

Litex 的工作就是把这种“看起来像普通代数推导”的东西变成可检查的事实链。每一行不是
给 checker 的操作指令，而是一条数学断言。断言通过以后，后面的行就能继续使用它。

再看一个稍微长一点的计算链：

```litex
forall a, b Q:
    a - b = 4
    a * b = 1
    =>:
        (a + b)^2 = (a - b)^2 + 4 * (a * b) = 4^2 + 4 * 1 = 20
```

这里真正发生了三件事：

1. 把 `(a + b)^2` 改写成 `(a - b)^2 + 4 * (a * b)`。
2. 用上下文里的 `a - b = 4` 和 `a * b = 1` 代入。
3. 做最后的数值计算。

这就是“用户写事实，Litex 增长上下文”的意思。它不是把证明隐藏起来，而是让证明以
数学事实的形状显出来。

## 先写你需要的数学形状

很多时候，证明卡住不是因为数学上完全不知道，而是因为形式化系统要求你先想清楚：
我现在应该用哪个 tactic？库里定理叫什么？这个目标在系统内部被编码成什么类型？

Litex 尽量把第一步改成另一件事：先写你需要的数学形状。

比如，“能被 8 整除”可以直接定义为“存在整数 `d`，使得 `x = 8 * d`”。要证明它能推出
“能被 2 整除”，不需要引入抽象谓词：从已有 witness `d` 取新 witness `4 * d` 即可。

```litex
prop can_be_divided_by_8(x Z):
    exist d Z st {x = 8 * d}

prop can_be_divided_by_2(x Z):
    exist d Z st {x = 2 * d}

claim:
    ? forall x Z:
        $can_be_divided_by_8(x)
        =>:
            $can_be_divided_by_2(x)
    obtain d from exist d Z st {x = 8 * d}
    witness exist e Z st {x = 2 * e} from 4 * d:
        x = 8 * d
        8 * d = 2 * (4 * d)

witness exist d Z st {8 = 8 * d} from 1:
    8 = 8 * 1
$can_be_divided_by_8(8)
$can_be_divided_by_2(8)
```

`obtain` 从“能被 8 整除”的定义中取出 `x = 8 * d` 的 witness；`witness` 再用
`4 * d` 构造 `x = 2 * (4 * d)`。最后三行把 `8` 作为具体例子放入这条已证明的规律。

这不是说 Litex 会自动理解所有数学。它做的是更窄的一件事：当你写出的目标事实和当前
上下文里的定义和事实形状吻合时，checker 尝试把它们匹配、展开或计算。等式、成员关系、
函数定义、全称事实、存在见证、分类讨论，都是这种风格的一部分。

## 失败也应该有用

一个形式化系统如果只告诉你“过了”或者“没过”，调试会很痛苦。Litex 希望失败也能变成
下一步行动的提示。

大体上，Litex 的反馈分三类。

`true` 表示这条事实已经被当前上下文证明了。你可以继续往下写。

`unknown` 表示这句话有意义，但现在的上下文还不够。通常下一步不是乱搜定理名，而是把
中间步骤拆小一点：补一个等式，补一个见证，补一个分类讨论，或者先写出一个更具体的
`forall`。

`error` 表示更基础的问题：语法不对、对象没有声明、函数调用不满足定义域、除法没有
证明分母非零，或者表达式本身还没有良好定义。

成功时，Litex 还可以解释它为什么成功。比如一条事实是通过引用已有的 `forall` 得到的，
输出里会记录被引用的全称事实、实例化方式，以及为了使用它需要检查的条件。你不只是知道
“这一行过了”，还可以看到它靠的是哪个上下文事实、哪个 builtin rule，或者哪个已知
`forall`。

这点对人有用，对 AI agent 也有用。因为局部反馈越清楚，修复证明就越像调试数学，而不是
盲目改文本。

## 集合、函数和普通数学对象

Litex 的表面语言故意靠近很多教材里默认的集合论写法。集合、元素、函数、元组、有限集、
set-builder，都尽量按普通数学对象出现。

例如成员关系：

```litex
1 $in {n Z : n <= 3}
2 $in {1, 2, 3}
not 4 $in {1, 2, 3}
```

`{n Z : n <= 3}` 表示所有满足 `n <= 3` 的整数。要证明 `1` 属于这个集合，Litex 检查的
就是把 `1` 代入条件后得到的 `1 <= 3`。

有限集合也可以直接枚举：

```litex
forall a {1, 2, 3}:
    a = 1 or a = 2 or a = 3
```

函数的定义域可以写在参数位置：

```litex
have fn g(x R: x > 0) R = x + 1

g(1) = 2
```

这里调用 `g(1)` 时，Litex 不只是展开 `g` 的定义，还要检查 `1` 满足输入条件
`x > 0`。

分段函数也可以按数学课本里的形状写：

```litex
have fn h(x R) R by cases:
    case x = 2: 3
    case x != 2: 4

h(2) = 3
```

这些设计背后的想法很朴素：很多数学证明的困难不在于对象本身多抽象，而在于形式化时过早
被迫处理编码细节。Litex 希望让读者先站在自然的数学层面上，把对象、条件和事实说清楚。

## 可以从合适的抽象层开始

数学家经常不是从最底层构造开始写证明的。做几何时，先说点、线、相交关系；做代数时，
先说运算和公理；做分析时，先说序、极限、邻域或度量。具体模型以后可以补，但一开始要
先把要研究的世界搭起来。

Litex 支持这种写法。你可以用 `have` 引入对象和集合，用 `abstract_prop` 引入暂时不展开
的原始关系，用 `prop` 定义派生概念，用 `axiom` 声明有名字的背景接口，用 `trust`
记录当前阶段临时当作背景假设的事实。

这并不表示所有东西都已经被证明了。`axiom` 是有名字的 trusted theorem-like interface；
`trust` 的意思也很明确：把一个事实作为显式假设放进上下文。它们都不是 Litex
在这一行证明出来的结论。

但这种抽象起步很重要。它让你可以先写出一段数学理论的接口，然后逐步把 `trust` 换成
真正检查通过的证明。对教材翻译、几何、公理化结构、分析接口和 AI 生成证明来说，这种
从抽象层开始的反馈循环很实用。

## 像翻译教材一样形式化数学

Litex 很关心的一件事，是形式化数学不要变成单纯“查库定理”。

在成熟 proof assistant 里，形式化一本教材时，一个很常见的路线是 import 一个大库，
找到库里已经证明好的同义定理或更强定理，然后引用它。这样当然很强，也很有工程价值。
Lean 和 Mathlib 的威力就在这里。

但如果目标是学习一段教材，或者把一本书的证明过程变成可检查材料，这种做法有时会把最
重要的部分移走。读者看到的是“这个定理在库里的哪里”，而不是“这本书的证明怎样一步步
走到这里”。

Litex 想服务的是另一种体验：像把英文数学书翻译成中文，或者把手写证明整理成 LaTeX。
你尽量顺着原文的数学结构往下写：先有定义，再有小 lemma，再有局部计算，再用前面已经
建立的事实证明后面的定理。

这就是 Litex 说“轻依赖”时真正想表达的意思。不是没有 trusted background，也不是所有
数学都从空逻辑重建；而是教材的主要数学劳动应该留在当前文件和前面章节里。import 可以
提供背景词汇和必要接口，但不应该把一本书的主体证明吞掉。

## 一个函数证明的样子

下面这个例子证明一个仿射函数 `p_affine(x) = 2 * x - 5` 是双射。它看起来长一些，但结构
还是普通数学：先证明单射，再证明满射，最后合成双射。

```litex
prop injective_fn(S, T set, f fn(x S) T):
    forall x1, x2 S:
        f(x1) = f(x2)
        =>:
            x1 = x2

prop surjective_fn(S, T set, f fn(x S) T):
    forall y T:
        exist x S st {y = f(x)}

prop bijective_fn(S, T set, f fn(x S) T):
    $injective_fn(S, T, f)
    $surjective_fn(S, T, f)

have fn p_affine(x R) R = 2 * x - 5

claim:
    ? forall x1, x2 R:
        p_affine(x1) = p_affine(x2)
        =>:
            x1 = x2

    x1 = ((2 * x1 - 5) + 5) / 2 = (p_affine(x1) + 5) / 2 = (p_affine(x2) + 5) / 2 = ((2 * x2 - 5) + 5) / 2 = x2

$injective_fn(R, R, p_affine)

claim:
    ? forall y R:
        exist x R st {y = p_affine(x)}

    witness exist x R st {y = p_affine(x)} from (y + 5) / 2:
        y = 2 * ((y + 5) / 2) - 5 = p_affine((y + 5) / 2)

$surjective_fn(R, R, p_affine)

$bijective_fn(R, R, p_affine)
```

这段证明的好处是每个失败点都比较局部。如果单射失败，就去看等式链是否太大、是否需要
拆中间步骤。如果满射失败，就检查 witness 是否合适。如果函数调用失败，就看定义域条件
有没有证明出来。

这就是 Litex 想要的 proof loop：不是一次写出完美证明，而是让每次失败都指向一个更小的
数学 gap。

## 和 Lean 的关系

Litex 不是“更快的 Lean”，也不是 Lean、Coq、Isabelle 的替代品。

Lean 有成熟的 kernel、Mathlib、生态、编辑器支持和社区。它适合严肃的大规模形式化数学
工程。Litex 现在更年轻，也更实验。它测试的是另一个假设：

> 一个更小、更可读、以事实为中心的形式化语言，能不能让学生、领域科学家和 AI agents
> 更便宜地产生有用的 checked mathematical data。

区别不在于 Lean 能不能证明这些小例子。Lean 当然能。区别在默认界面。

在 Lean 里，你常常是在管理 goal state：选择 tactic，引用定理，处理类型和库约定。
在 Litex 里，你更常做的动作是写下一条数学事实，让 checker 看它是否能从当前上下文推出。

反馈循环。

## 也要把边界说清楚

Litex 现在仍然是研究型、实验型系统。它不应该被宣传成已经成熟的 production formal
verification 系统。

几个边界需要一直放在明面上：

- `trust` 是 assumption injection。它把事实作为显式假设加入上下文，不是证明。
- builtin rule 和 infer rule 属于 trusted mathematical background，需要文档、测试和审计。
- 当前 examples 和教材片段是 prototype evidence，不是 production-ready 证明库。
- AI 生成的 Litex 代码必须看 verifier output、剩余 `trust` 和失败原因，不能只看文本是否像证明。
- Litex 的 trusted implementation 比成熟小核系统更大。这个取舍必须靠清楚的文档、测试、
  verification 输出和规则边界来支撑。

所以，最稳妥的理解不是“Litex 已经达到了 Lean 那种成熟形态”，而是：
Litex 有意识地用一个更直接、更关系优先的 checker，去换取更自然的数学表面和更短的
checked feedback loop。这个实验是否成功，要靠可运行例子、失败记录、测试和审计来说话。

## 目前最值得看的材料

Litex 现在最硬的资产，不是一个漂亮的 toy example，而是一批可以运行、可以回归测试的
examples、标准库片段和数据集翻译尝试。

其中最重要的一条线是 The Mechanics of Litex Proof。它把 Heather Macbeth 的
The Mechanics of Proof 中的 Lean 训练材料改写成 Litex proof style，形成了连续章节和
两百多个可运行片段。读者可以看到 Litex 怎样处理等式、不等式、全称、存在、反证、分类、
归纳、数论、函数、集合、关系和基数这些基础证明训练。

这套材料的价值，不是某个 theorem 被一行关掉了，而是整条学习路径留下来了。定义怎样出现，
lemma 怎样积累，失败点怎样缩小，checker 怎样给出反馈，这些都比单个结论更重要。

MATH500、MiniF2F-style 任务、高中数学题、Mathematics in Lean、Tao Analysis 等翻译尝试，
则继续作为压力测试。能通过的题目变成 example 或 benchmark；不能通过的题目也有价值，
因为它们会暴露缺少的定义、lemma、builtin rule、infer rule、语法能力或诊断能力。

换句话说，Litex 不只是想展示“模型会写几个像证明的东西”。它想把普通数学推理放进一个
机器可检查、可解释、可迭代的工程循环里。

## 最后再说一遍

Litex 的核心不是“语法短”，而是一个更直接的数学反馈界面。

你写下一条事实。Litex 按它的形状去匹配当前上下文、已知规则和基础数学关系。检查通过，
这条事实就进入 verified context；检查失败，输出应该帮助你看见是数学 gap、定义良好性问题，
还是上下文还不够。

如果读完这篇只记住一句话，可以记这句：

> Litex 想让普通数学证明变成一条可以运行、可以解释、可以一步步修复的事实链。

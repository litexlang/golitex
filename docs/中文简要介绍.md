## Litex 是什么

Litex 是一个面向数学推理检查的开源形式化语言和验证系统。它的目标不是把数学写成只有少数专家能读的底层证明对象，而是尽量保留普通数学写作的外观，让用户仍然按照接近教材、讲义、草稿纸的方式写出数学对象、数学事实和证明结构，然后由系统逐步检查这些步骤是否成立、是否定义良好、是否真的能从当前上下文推出。

一句话说，Litex 想做的是：

> 让“像人写的数学推理”进入“机器可检查”的反馈回路。

这件事对 AI 尤其重要。今天的大模型已经能生成很多数学推导、证明草稿和公式解释，但真正难的不是“写出来”，而是“逐步检查它到底哪里对、哪里错、哪里缺条件”。Litex 试图提供的，正是这种中间层：既不是纯自然语言，也不是完全脱离普通数学写法的底层证明脚本，而是一种可以逐句检查的数学语言。

## Litex 想解决什么问题

很多数学推理错误并不是结论完全荒谬，而是局部失真：

- 某一步变形其实不合法；
- 忽略了定义域条件；
- 把“对某个变量成立”误当成“对所有变量成立”；
- 用了一个并未说明的已知事实；
- 图形直觉、文本直觉、代数直觉之间偷偷跳步；
- 证明结构本身看起来对，但缺了关键连接环节。

对人类读者来说，这些错误有时可以靠经验识别；对 AI 系统来说，如果没有一个能逐步返回验证结果的环境，这些错误很难被稳定地发现和修复。

Litex 的定位正是在这里：它不直接替代成熟 theorem prover，也不只是一个“数学 DSL”。它更像一个为数学推理提供局部可验证反馈的执行环境。用户或 AI 写下一串数学语句，Litex 解析、执行、验证，并返回结构化结果：哪些语句成功，成功是因为 builtin rule、已知事实还是已知的 `forall` 事实；哪些语句是未知；哪些语句因为对象未定义或语法不合法而直接报错。

## Litex 的核心设计观

Litex 的核心设计观可以概括为一句话：

> 用户写事实，系统增长一个可解释的已验证上下文。

这和很多传统 proof assistant 的使用体验不同。Litex 更强调以下几点：

1. 数学事实是第一公民。  
   用户主要写的是等式、不等式、成员关系、函数关系、存在与全称、分类讨论、归纳结构这些数学陈述本身，而不是大量 tactic 命令。

2. 上下文是累积增长的。  
   一条事实一旦被验证通过，就会进入当前上下文，后续步骤可以继续引用、匹配、实例化和推断它。

3. 系统承担一部分例行数学工作。  
   很多普通数学证明里的机械环节，例如代数归一化、已知等式代入、集合成员关系拆分、函数定义域检查、`forall` 实例化等，由 checker 内建完成。

4. 输出必须可解释。  
   Litex 不只是返回“通过/失败”，而是尽量说明一条事实是如何成立的。这一点对 AI agent workflow 很关键，因为 agent 需要的不只是 verdict，还需要局部修复线索。

## Litex 代码看起来像什么

Litex 追求一种接近普通数学写法的形式。例如：

```litex
forall x R:
    x = 2
    =>:
        x + 1 = 3
        x^2 = 4
```

这段代码表达的是一个很普通的数学结构：给定实数 `x`，如果 `x = 2`，那么推出两个结论。Litex 的目标不是把这个证明写得炫技，而是让这个结构本身就可以被检查。

这背后的思想是：很多教材级、应用数学级、课堂证明级的推理，本来就是围绕对象、条件和结论的组织关系展开的。Litex 试图保留这种阅读顺序和叙述顺序，而不是要求用户首先学习一套独立于数学内容之外的战术语言。

## 系统实现上，Litex 是怎么工作的

从实现角度看，Litex 目前是一个以 Rust 为核心的验证系统。仓库中的 `src` 目录已经清楚分出了几个核心层次，例如：

- `parse`：负责把源码解析成语句和对象；
- `runtime`、`environment`、`module_manager`：负责当前运行时上下文、作用域、文件路径和模块信息；
- `verify`：负责具体的事实验证；
- `infer`：负责在已验证事实的基础上自动推出标准后果；
- `execute`：负责执行语句，让上下文真正发生变化；
- `pipeline`：负责把“读源码 -> 解析 -> 执行 -> 输出结果”串起来；
- `result`、`error`：负责结构化结果和错误输出。

更具体地说，当前的一次 Litex 运行大致经历下面几步。

### 1. 读取源码并切分 block

在 `pipeline::run_source_code` 中，源码首先被 tokenizer 切成 block，然后逐个 block 解析成内部语句表示。这里的基本思想不是一次性把整份文件变成一个静态 AST 后再统一证明，而是按脚本执行的顺序逐步处理。

### 2. 在 Runtime 中逐句执行

Litex 文件不是单纯的声明列表，而是一个会改变上下文的数学脚本。每条语句都可能：

- 引入对象；
- 断言事实；
- 开启证明块；
- 注册定义；
- 把已验证事实存入上下文；
- 触发内建推理。

也就是说，Litex 不是只检查单条定理，而是在运行时维护一个“当前数学世界”。

### 3. 验证事实是否成立

当用户写下一条事实时，Litex 会检查：

- 这条事实是否语法正确；
- 里面的对象是否定义良好；
- 是否满足函数定义域、除零约束、类型/集合约束等前提；
- 能否通过 builtin verification rules、已有事实、已有 `forall` 事实、局部证明结构等方式得到证明。

这一层是 Litex 的核心。当前系统已经内置了不少普通数学里高频出现的验证能力，尤其是数值计算、基本代数关系、集合和成员关系、有限对象结构、一些常见的函数空间与定义域事实等。

### 4. 存储通过的事实并自动推理

Litex 和很多“只判定当前句子真伪”的小系统不同，它会把验证通过的事实放入上下文，然后触发一批标准推理。例如：

- 从某些等式推出对象的值绑定；
- 从集合交中的成员关系推出分别属于两个集合；
- 从函数、笛卡尔积、元组等对象的已知关系中推出结构性后果；
- 从数值顺序关系推出符号信息；
- 从已知 `forall` 事实实例化出更具体的可用事实。

这也是 Litex 能把证明脚本保持得比较短的原因之一：用户不需要手工重复所有显然的中间步骤。

### 5. 返回结构化输出

Litex 的输出不是一行简单的 “OK” 或 “FAIL”。当前的 JSON 风格输出会尽量说明：

- 当前语句的类型；
- 对应的源码行；
- 最终结果是 success、unknown 还是 error；
- 如果成功，是由 builtin rule 证明、还是引用了已知事实、还是引用了某个 `forall` 事实；
- 推理过程中自动得到的 `infer_facts`；
- 如果失败，具体是哪里出错。

这使它不只是一个 verifier，也是一个适合人和 AI 共同调试证明的反馈系统。

## Litex 为什么特别适合 AI agent workflow

如果把 Litex 仅仅理解成“又一种形式化语言”，会低估它的价值。它真正有潜力的地方，在于它和 agent 的交互方式非常自然。

一个大模型在 Litex 环境中的典型工作流是：

1. 先把题目或命题写成自然语言解法；
2. 再把每一步翻译成 Litex 语句；
3. 运行 verifier；
4. 读取哪一步 unknown、哪一步 error、哪一步因为条件不够而失败；
5. 继续补局部引理、补条件、改语句结构；
6. 重复迭代，直到整段证明可检查。

这和直接让模型一次性吐出一个“最终形式化证明”不同。Litex 提供的是局部调试界面。模型不必一开始就拥有完整 theorem library 视图，而是可以在反馈回路里逐步逼近正确证明。

这也是为什么 Litex 当前非常适合做以下几类工作：

- 教材级证明 formalization 的 agent 闭环；
- 数据集题目的局部可验证转换；
- AI for Science 中公式推导片段的检查；
- 作为自然语言数学推理和成熟 theorem prover 之间的中间层。

## 现在已经做到了什么

Litex 目前还处在原型到系统化工程之间的阶段，但已经不是一句概念口号。现有仓库和文档显示，它已经具有以下比较明确的原型信号：

- 有可运行的 Rust 内核；
- 有官网和在线 playground：https://litexlang.com
- 有成体系的 Manual、Cheatsheet 和其他说明文档；
- 有教材风格的 The Mechanics of Litex Proof：https://litexlang.com/doc/The_Mechanics_of_Litex_Proof
- 有大量 `examples/*.lit`；
- 有面向 Python 的 `litexpy` 方向，用于把 Litex 放进脚本和 agent workflow；
- 有成规模的测试入口，用来同时跑 examples、Manual 代码块和 Mechanics 教材中的代码块；
- 已经能够支撑一些非玩具的教材证明案例，而不只是 `1 + 1 = 2` 级别的演示。

这说明 Litex 已经跨过了“只在脑子里成立”的阶段，进入了“可以持续验证设计、可以观察系统行为、可以让外部人复现实验”的阶段。

## 但 Litex 现在还不是什么

为了避免误解，也需要明确说出 Litex 目前还不是什么。

它目前还不是：

- 一个生产级 theorem prover；
- 一个已经完成 soundness 论证和 trusted base 规约的成熟内核；
- 一个拥有大规模现成数学库的系统；
- 一个可以直接替代 Lean、Coq、Isabelle 的工业级工具链；
- 一个已经完成 benchmark、性能、生态和社区稳定建设的平台。

换句话说，Litex 的当前价值主要在于：

- 它提出了一种和现有主流 proof assistant 不同的形式化接口；
- 它已经做出了足够多的实现，让这个接口可以被真实测试；
- 它对 AI agent feedback 这一问题给出了一条值得继续走的工程路线。

## 当前最值得认真做的工程方向

如果把 Litex 当作一个长期项目看，当前最重要的方向不是继续堆宣传，而是继续把以下几件事做深。

### 1. 讲清 soundness boundary

系统里哪些能力属于 trusted kernel，哪些属于 builtin rule，哪些是可接受的临时设计，哪些还只是实验性能力，需要明确写清楚。否则系统越大，外部越难判断它的证明结果到底可信到什么程度。

### 2. 扩基础数学库和背景规则

如果 Litex 想真正支持更多教材级和应用级推理，就必须逐步建设稳定的背景数学知识层，包括集合、函数、数论、离散数学、初等代数、分析里常见局部推导结构等。

### 3. 做可复现 benchmark

只有把 examples、教材案例、题目数据集和 agent formalization 流程变成可复现 benchmark，外部人才有办法判断 Litex 的进展到底是“主观感觉更好”还是“客观能力更强”。

### 4. 强化与 agent 的闭环

Litex 的独特价值很大一部分来自 agent workflow，因此需要继续优化：

- verifier 输出的可读性；
- 局部错误定位；
- Python / API 接口；
- 批量运行和结构化结果收集；
- 从失败中生成修复建议的上层工具。

### 5. 把 AI for Science 示例做成真正案例

很多科学推导并不需要立刻进入最重型的 theorem prover 生态，但它们确实需要一个局部可检查的数学反馈环境。Litex 很适合先从这些“中等正式度”的场景切入，做出真正能用的案例。

## 为什么这个方向值得继续

今天大家已经见过很多“AI 会做数学”的展示，但真正缺的不是 demo，而是反馈基础设施。

如果一个系统能够让 AI 生成的数学推理变成：

- 可逐步检查；
- 可定位错误；
- 可积累上下文；
- 可重复运行；
- 可由人和机器共同迭代修复；

那它就不只是一个语言实验，而是在建设 AI 数学推理和科学推导的基础设施层。

Litex 最值得认真看待的地方，恰恰不是它已经完成了一切，而是它已经把一个重要问题做成了可以被持续工程化的对象：如何让普通数学写法进入机器可检查、可解释、可反馈的执行环境。

## 结语

Litex 现在最合理的理解方式，不是“一个已经成熟的证明系统”，也不是“一个漂亮但空的概念”。它更像一个已经跑起来的实验性基础设施项目：方向明确，系统轮廓清楚，关键实现已经出现，真实价值要靠后续工程和 benchmark 继续证明。

如果这个方向继续做下去，Litex 可能会成为 AI 数学推理、教材 formalization、科学推导检查之间的一层关键中间层。它未必要替代所有现有系统，但它完全可能在“让普通数学推理变成可验证反馈对象”这件事上，建立出自己独立而重要的位置。
